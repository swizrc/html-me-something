The Problem

The main motivation for me making this app was to help my girlfriend meet her goals by providing her a convient and comprehensive application for tracking her caloric and protein intake.

I sought to make something that was more dynamic and fun to use from a mobile perspective. However, having done zero mobile development, I had to start learning a completely different development platform than anything I had seen before.

The Platform

I had a choice between two major mobile platforms, iOS and Android. My girlfriend uses an iPhone so that made my choice clear.

To develop an app for iOS, Apple expects a developer to own a Mac computer and use their proprietary IDE called Xcode.

Seeing as I do not own anything manufactured by Apple, I had to go through a process of obtaining a disk image of OSX and install it for use with a virtual machine using VirtualBox.

With Xcode installed and ready for use on my Windows PC, I then had to learn a brand new language.

The Language

Since the launch of iOS, iOS applications have been developed using Objective C. However, within the past few years, Apple developed a new programming language of their own, called Swift®.

I chose to learn Swift over Objective C as I thought the syntax was a lot easier to learn and also to take advantage of the new features in Swift over Obj. C, such as optionals.

By nature, mobile development is heavily based on object-oriented programming. While I have had some experience with OOP in the past with C++, my experience was limited before tackling this project.

Learning Swift and it's intricacies has been a very rewarding endeavour, but also a very difficult one; probably far more difficult than the entire scope of LC101.

While the Foodtracker app is still in progress, the standard framework is already established.

The Front-End

Designing the front end user interface is fairly intuitive. One could programmatically create all the view and elements an application would have in theory, but that would be a huge waste of time.

A big part of Swift is called The Storyboard. The user interface can be designed here using drag and drop elements, as well as establishing connections between these elements for dynamic interations.

The nodes seen comprising this storyboard as the views that the user sees.

In my application, the user will mostly see tables of elements that they will tap on to make a selection, such as selecting which user they are, which date they want to modify food for, viewing a certain food's details, etc.

While much of the front end can be made with the storyboard, I had to create a lot of back end programming that makes the backbone of the Foodtracker app.

The Back-End

The back end of the app has a huge amount of elements to it, many of them I have never encountered before this project.

Much of the new concepts has to do with interfacing Swift code with the elements created in the storyboard. One must create something called "outlets" connected to storyboard elements one wants to manipulate in code.
For example, if I want to change some text in a text label element, I need to let the back end know that such a label exists before I change it's text.

All views on the storyboard are their own encapsulated class. The views do not know of each other and must pass data between them through transistions, known as segues.

Passing data forwards and backwards from views is an essential part in the operation of this app's data handling as every view will process the data in a different meaningful way.

Much work has been done on this app, but there is still much more to be done. I have shared just an entry in my journey of developing my first iOS application, dedicated to someone I love very dearly.
